# Config Reference (`Config.h`)

## Debug logging

| Define | Default | Notes |
|---|---|---|
| `ESP_COMMANDER_DEBUG` | defined | An empty feature-flag macro (no replacement text) — its only purpose is to be tested with `#ifdef`. Undefine it before including the library to compile out `ESP_COMMANDER_LOG` entirely. |
| `ESP_COMMANDER_LOG(x)` | — | Not user-configurable directly; expands to `Serial.println(x)` when `ESP_COMMANDER_DEBUG` is defined, and to nothing otherwise. Used throughout the library for internal diagnostics. |

:::caution
`ESP_COMMANDER_DEBUG` is a plain flag, not a logging macro — don't call it like a function (`ESP_COMMANDER_DEBUG("message")`). A few places in `SleepyDevice.h` currently do exactly that; since the macro has no parameter list, the call site is left as a harmless-but-silent discarded string expression rather than a log line. See [Troubleshooting](../troubleshooting) for the full list of affected call sites.
:::

## Overridable limits

All of the following are guarded with `#ifndef`, so redefine them **before** `#include "EspCommander.h"` if you need different limits:

| Define | Default | Purpose |
|---|---|---|
| `ESP_COMMANDER_SCHEMA_QUERY` | `"__SCHEMA__"` | The reserved query name that triggers `Device::schema()` instead of a lookup against registered queries. Avoid naming a real query this unless you intentionally override the define. |
| `ESP_COMMANDER_JSON_BUFFER_SIZE` | `8192` | Suggested size for the buffers you pass to `StaticBufferAllocator`. Not enforced by the library — it's a convention, you choose the actual buffer size at the call site. |
| `ESP_COMMANDER_MAX_PARAMETERS` | `8` | Size of the `HandlerValue[]` stack array `Device::request()` / `SleepyDevice::action()` use to pass parameters to a handler. Any single action declaring more parameters than this will read/write out of the usable range. |
| `ESP_COMMANDER_MAX_RESULTS` | `8` | Same as above, for results, used by `Device::request()` and `SleepyDevice::data()`. |
| `ESP_COMMANDER_MAX_QUERIES` | `8` | Not currently referenced by array sizing in the shipped headers — treat as a documented convention for how many queries a `Device` is expected to register, useful for sizing your own `Query[]` array. |
| `ESP_COMMANDER_MAX_ACTIONS` | `8` | Same convention as above, for actions. |
| `ESP_COMMANDER_MAX_ENUM_SIZE` | `8` | Documented convention for how many entries an `ENUM` `Value`'s `enumDefinition` is expected to hold; not enforced by `Value` itself, which accepts whatever `etl::span` you give it. |

```cpp
#define ESP_COMMANDER_JSON_BUFFER_SIZE 4096
#define ESP_COMMANDER_MAX_PARAMETERS 4
#define ESP_COMMANDER_MAX_RESULTS 4

#include "EspCommander.h"
```
